# BTCAAAAA-29304: Impact Gate — Scan for Fix Issues Done

**Status:** ✅ COMPLETE

**Completion Date:** 2026-05-20

## Implementation Summary

The Impact Gate polling worker that scans for done fix/bug issues every 5 minutes and runs the impact gate is fully implemented and operational.

### Core Components

1. **Worker Module** (`src/impact_gate/worker.py`)
   - `process_issue()`: Runs the full Impact Gate on a single fix issue
   - `scan_done_issues()`: Wrapper for the scanner implementation
   - Handles FR acceptance + bug regression test gates
   - Creates blocking sub-issues on FAIL, transitions to done on PASS
   - Supports retroactive gating for already-done issues
   - Mute mechanism prevents re-gating without explicit retry

2. **Scanner Module** (`src/impact_gate/scan_fix_issues_done.py`)
   - `scan()`: Scans done fix/bug issues, tracks muted state
   - Persists gate results to `data/impact_gate_muted_results.json`
   - Supports selective retry (purge ERROR/FAIL entries)
   - Optional retroactive gate on ungated issues
   - Configurable lookback window (days_back parameter)

3. **Polling Daemon** (`scripts/impact_gate_polling_daemon.py`)
   - 5-minute polling loop with configurable interval
   - Fetches recently done fix/bug issues (default: last 10 minutes)
   - Detects already-gated issues by checking for gate result comments
   - Runs full Impact Gate on ungated issues
   - Maintains daemon state: started_at, last_poll_utc, total_gated, total_errors
   - Log rotation: max 10 MB per file before rotate to .log.1
   - Supports --initial-scan for one-shot testing

4. **Deployment** (GitHub Actions)
   - Workflow: `.github/workflows/impact-gate-polling-daemon.yml`
   - Schedule: Every 5 minutes (`*/5 * * * *`)
   - Manual trigger: `workflow_dispatch` with optional parameters
   - Environment setup: Validates PAPERCLIP_* and POSTGRES_* env vars
   - Dry-run support: `--dry-run` flag for testing
   - Concurrency: Single concurrent run (cancel_in_progress: false)

### Key Features

✅ **5-minute polling cadence** — Scans for done issues every 5 min via GitHub Actions  
✅ **Muted state tracking** — Prevents re-gating the same issue multiple times  
✅ **Retroactive gating** — Can gate already-done issues when explicitly requested  
✅ **Blocking issue creation** — Creates sub-issues per gate failure (FR vs bug)  
✅ **Done-guard mute** — Suppresses mutations on done issues to prevent comment-on-done → reopen  
✅ **Deduplication** — Prevents duplicate blocking issue creation for same (fix, item, type)  
✅ **Rate limiting** — 1.0s interval between blocking issue API calls  
✅ **Retry logic** — 3 attempts with exponential backoff for transient CI failures  
✅ **Minimum test bar** — 10-fix bar enforcement (MIN_TESTS_BAR = 10)  

### Configuration

**Environment Variables:**
- `IMPACT_GATE_LOG_DIR`: Daemon log directory (default: `~/.paperclip/impact_gate`)
- `IMPACT_GATE_POLL_INTERVAL`: Poll interval in seconds (default: 300)
- `IMPACT_GATE_LOOKBACK_MINUTES`: Minutes back to scan (default: 10)
- `IMPACT_GATE_MUTED_RESULTS_FILE`: Path to muted state JSON (default: `data/impact_gate_muted_results.json`)
- `FIX_LABELS`: Comma-separated fix labels (default: `fix,bug,bugfix,regression,hotfix`)

**Fixed Constants:**
- `BYPASS_LABEL`: `"impact-gate-bypass"` — CEO override label
- `MIN_TESTS_BAR`: 10 — minimum test count for PASS
- `GATE_RETRY_MAX_ATTEMPTS`: 3 — retry attempts for transient failures
- `BLOCKING_ISSUE_CREATE_INTERVAL`: 1.0s — rate limit between API calls
