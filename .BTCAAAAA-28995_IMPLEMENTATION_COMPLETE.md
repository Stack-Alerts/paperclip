## Implementation Details

### 1. Health Check Script (`scripts/paperclip_recovery_health_check.py`)

**Purpose:** Periodically validates the recovery monitor is operational.

**Checks Performed:**
- **Monitor Execution**: Verifies last run timestamp is recent (< 45 min old)
- **Configuration Validity**: Validates recovery scenarios, agent scopes, and defaults
- **State Health**: Checks recovery metrics, escalations, and recovery loops
- **Log Health**: Analyzes for error rates and fatal issues
- **Systemd Timer**: Confirms backup scheduler is running (if applicable)

**Health Status Levels:**
- `healthy` — All checks pass
- `degraded` — One or more warnings (non-fatal issues)
- `unhealthy` — One or more check failures (requires immediate action)

**Usage:**
```bash
python scripts/paperclip_recovery_health_check.py --json-report
python scripts/paperclip_recovery_health_check.py --alert-on-unhealthy
```

### 2. Stalled Workflow Reporter (`scripts/paperclip_stalled_workflow_reporter.py`)

**Purpose:** Analyzes and reports on workflows currently in recovery.

**Reports:**
- Issues undergoing recovery by scenario
- Recovery attempt counts and age
- Escalated issues requiring manual intervention
- Issues stuck in rapid recovery loops

**Usage:**
```bash
python scripts/paperclip_stalled_workflow_reporter.py --json-report
python scripts/paperclip_stalled_workflow_reporter.py --scenario exchange_api_timeout
```

### 3. GitHub Actions Workflow (`.github/workflows/paperclip-recovery-health-check.yml`)

**Schedule:** Every hour (`:00` minute mark UTC)

**Steps:**
1. Run health check script → JSON report
2. Generate stalled workflow report → JSON report
3. Render combined summary to GitHub step summary
4. Upload reports and logs (30-day retention)

**Manual Trigger:**
```bash
gh workflow run paperclip-recovery-health-check.yml \
  -f alert_on_unhealthy=true
```

## Known Limitations

1. **Logging to stderr**: Both scripts log to stderr in addition to file handlers. This is intentional for troubleshooting but results in mixed console output. JSON output should be captured with stderr suppression: `python script.py --json-report 2>/dev/null`

2. **No alerting automation**: Health check identifies problems but doesn't automatically page/alert. Integration with external alerting systems is recommended.

3. **SystemD timer check is optional**: Systemd checks gracefully skip if systemd is unavailable (OK for GitHub Actions-only deployments).
