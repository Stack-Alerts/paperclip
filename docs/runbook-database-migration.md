**Owner:** DatabaseAdministrator / DocWriter
**Audience:** Database Administrators, Platform Engineers, Developers

## 1. Overview

Database migrations use **Alembic** with SQLAlchemy models as the source of truth. The migration system is configured in `alembic.ini` and `alembic/env.py`.

### Key Files

| File | Purpose |
|------|---------|
| `alembic.ini` | Alembic configuration (script location, logging) |
| `alembic/env.py` | Runtime configuration: connects to DB via `get_db_url()`, auto-detects model changes |
| `alembic/versions/` | Migration revision files (one per change) |
| `alembic/script.py.mako` | Migration template |
| `src/optimizer_v3/database/models.py` | SQLAlchemy `Base` metadata — source of truth for autogenerate |
| `scripts/manage_migrations.py` | Migration management CLI |
| `scripts/manage_backups.py` | Backup/restore CLI |

### Database Configuration

The database URL is resolved at runtime by `src.optimizer_v3.database.config.get_db_url()`, which reads from `.env`:

```ini
POSTGRES_HOST=localhost
POSTGRES_PORT=5432
POSTGRES_DB=optimizer_v3
POSTGRES_USER=optimizer_admin
POSTGRES_PASSWORD=<your_password>
```

## 2. Migration Script

All migration operations go through `scripts/manage_migrations.py`:

### 2.1 Create a Migration

```bash
# Auto-detect model changes (compare models.py vs database state)
python scripts/manage_migrations.py create "Add user_preferences table"

# Create empty migration (manual SQL)
python scripts/manage_migrations.py create "Custom index migration" --no-autogenerate
```

**What happens:**
1. Alembic compares `Base.metadata` (from `models.py`) against the current database schema
2. Generates a migration script in `alembic/versions/` with upgrade/downgrade functions
3. The script is **not applied** — it must be reviewed and then explicitly upgraded

### 2.2 Review the Generated Migration

```bash
# View the generated file
ls -lt alembic/versions/ | head -5

# Preview the SQL without applying
alembic upgrade head --sql 2>&1 | less
```

**Always review autogenerated migrations** — Alembic may miss renames, detect false positives, or generate incorrect column type changes.

### 2.3 Apply Migrations

```bash
# Apply all pending migrations to head
python scripts/manage_migrations.py upgrade

# Apply to a specific revision
python scripts/manage_migrations.py upgrade a1b2c3d4e5f6
```

The script prompts for confirmation showing the target database host/port/database.

### 2.4 Rollback Migrations

```bash
# Rollback 1 step
python scripts/manage_migrations.py downgrade

# Rollback 3 steps
python scripts/manage_migrations.py downgrade 3
```

### 2.5 Check Migration State

```bash
# Show current revision (what's applied)
python scripts/manage_migrations.py current

# Show full history
python scripts/manage_migrations.py history
```

### 2.6 Raw Alembic Commands

For advanced operations not covered by the helper script:

```bash
# Preview upgrade SQL
alembic upgrade head --sql

# Check migration state
alembic current

# View history with verbose details
alembic history --verbose

# Show pending migrations
alembic check

# Stamp a specific revision (without running migrations)
alembic stamp a1b2c3d4e5f6
```

## 3. Migration Workflow

### 3.1 Standard Migration Flow

```
1. Edit SQLAlchemy models (src/optimizer_v3/database/models.py)
2. Create migration:      python scripts/manage_migrations.py create "description"
3. Review migration:      cat alembic/versions/20260513_1200_description.py
4. Test on dev DB:        python scripts/manage_migrations.py upgrade
5. Rollback test:         python scripts/manage_migrations.py downgrade 1
6. Upgrade again:         python scripts/manage_migrations.py upgrade
7. Commit migration:      git add alembic/versions/ && git commit
8. Backup prod DB:        python scripts/manage_backups.py create --compress
9. Apply to prod:         python scripts/manage_migrations.py upgrade
10. Verify:               python scripts/manage_migrations.py current
```

### 3.2 Safe Migration Principles

1. **Always backup before prod migration** — `python scripts/manage_backups.py create --compress`
2. **Test rollback** — verify `downgrade` works before deploying
3. **One logical change per migration** — avoid mixing schema and data changes
4. **Avoid irreversible migrations** — if you must, document in the migration file
5. **Lock module exceptions** — `src/optimizer_v3/database/` is a locked module; see [runbook-module-lock.md](runbook-module-lock.md)

## 4. Alembic Configuration Details

From `alembic/env.py`:

- **Metadata source:** `src.optimizer_v3.database.models.Base.metadata`
- **URL resolution:** `src.optimizer_v3.database.config.get_db_url()` (reads from `.env`)
- **Autogenerate options:** `compare_type=True`, `compare_server_default=True`
- **Logging:** SQLAlchemy at WARN, Alembic at INFO (configured in `alembic.ini`)
- **Naming convention:** `%(year)d%(month).2d%(day).2d_%(hour).2d%(minute).2d_%(rev)s_%(slug)s`

## 5. Common Issues

| Issue | Cause | Fix |
|-------|-------|-----|
| `Target database is not up to date` | Migration not applied on this DB | Run `python scripts/manage_migrations.py upgrade` |
| `Can't locate revision identified by X` | Missing migration file (branch divergence) | `git pull` or check `alembic/versions/` |
| Autogenerate produces empty migration | No detectable model changes, or models out of sync | Check `Base.metadata` imports, verify DB state |
| `FAILED: No support for ALTER of CREATE TABLE` | Alembic limitation for certain DB changes | Write manual `op.execute()` in migration |
| `psycopg2.OperationalError: Connection refused` | PostgreSQL not running or wrong host/port | Check `.env` and PostgreSQL service status |
| Migration applies locally but fails in CI | Ephemeral CI Postgres may have different state | Check `test.yml` service definitions |
| Lock gate blocks migration PR | `src/optimizer_v3/database/` is locked | Request exception via [runbook-module-lock.md](runbook-module-lock.md) Path A |

## 6. Migration File Anatomy

Each migration in `alembic/versions/` contains:

```python
"""Add user_preferences table

Revision ID: a1b2c3d4e5f6
Revises: 9f8e7d6c5b4a
Create Date: 2026-05-13 12:00:00.000000
"""
from alembic import op
import sqlalchemy as sa

revision = 'a1b2c3d4e5f6'
down_revision = '9f8e7d6c5b4a'

def upgrade() -> None:
    op.create_table('user_preferences',
        sa.Column('id', sa.Integer(), nullable=False),
        sa.Column('user_id', sa.Integer(), nullable=False),
        sa.Column('preferences', sa.JSON(), nullable=True),
        sa.PrimaryKeyConstraint('id')
    )

def downgrade() -> None:
    op.drop_table('user_preferences')
```

## 7. Emergency Migration Rollback

```bash
# 1. Rollback the database
python scripts/manage_migrations.py downgrade 1

# 2. If migration is irreversible, restore from backup
python scripts/manage_backups.py restore /path/to/backup.sql.gz --drop --yes

# 3. Re-apply AI grants (backup may predate grant fix)
python -c "from src.optimizer_v3.database.backup import get_backup_manager; get_backup_manager().reapply_ai_grants()"

# 4. Revert the code change
git revert HEAD
git push origin main
```

## 8. Safety Checklist

Before every migration apply:

- [ ] Backup taken: `python scripts/manage_backups.py create --compress`
- [ ] Target database confirmed: check `alembic current` first
- [ ] Migration reviewed: auto-generated SQL inspected for correctness
- [ ] Production window confirmed (if prod): no concurrent writers
- [ ] Rollback plan ready: migration downgrade is reversible
